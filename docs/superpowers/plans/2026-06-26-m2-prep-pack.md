# M2 Prep-Pack Generation Implementation Plan

> **For agentic workers:** Executed under the Claude↔Codex 互评 workflow — Codex implements each task via `codex:codex-rescue` (--write, run_in_background); Claude runs the tests Codex can't, reviews, and commits.

**Goal:** From an ingested thesis's evidence, have the LLM generate a prep pack (digest, key numbers, Q&A, hostile questions, theory/citation cards) as `prep_item` rows — each **bound to the evidence_unit(s) it cites and run through the M0d validator before it can be `verified`**. This is the hybrid content layer (spec §10): generation is assistive, the validator is the gate; no model say-so becomes `verified`.

**Architecture:** A pure prompt+schema layer (`lib/llm/prep-pack.ts`) defines the zod shape the LLM returns and builds a prompt that hands the model the evidence chunks *with their ids* and demands every item cite `evidence_unit_ids`. A `generation_run`-driven service (`lib/llm/prep-pack-run.ts`) calls the injectable `LlmClient` (M0c), then for each generated item: insert the `prep_item` (stamped with `generation_run_id`), bind its cited evidence (same-thesis enforced, M0d repo), and validate it (M0d `validatePrepItem` → `applyValidation`). Tests use the `MockLlmClient` — deterministic, no network.

**Tech Stack:** TypeScript (strict) · zod · better-sqlite3 · M0c `LlmClient`/`MockLlmClient` · M0d validator · M1 evidence. 

**Spec:** `docs/superpowers/specs/2026-06-23-D-viva-assistant-agent-generic-design.md` §10 (prep-pack), §6 (`prep_item`, `generation_run`), §8 (LLM), §11 (evidence binding).

**Scope:** generation schema/prompt, `generatePrepPack` (LLM call), `generation_run` repository, the orchestrated generate→persist→bind→validate run. **Out of scope:** examiner/judge (→ M3), the UI/progress (→ M4), regeneration/diffing, streaming, L4 LLM-advisory validation (a separate later refinement).

## Contracts

```ts
// What the LLM returns per item. evidence_unit_ids cite the ingest evidence the item is grounded in.
export const GeneratedItemSchema = z.object({
  type: z.enum(["digest", "key_number", "qa", "hostile", "theory_card", "citation_card"]),
  title: z.string().min(1),
  claim_text: z.string().min(1),
  evidence_quote: z.string().nullable().default(null),
  value_numeric: z.number().nullable().default(null),
  unit: z.string().nullable().default(null),
  evidence_unit_ids: z.array(z.string()).min(1), // must cite >=1 ingest evidence_unit
});
export const PrepPackSchema = z.object({ items: z.array(GeneratedItemSchema) });
export type GeneratedItem = z.infer<typeof GeneratedItemSchema>;
```

**The schema is intentionally lenient on unknown keys** (plain `z.object`, NOT `.strict()`; Codex P2-1): a stray extra field from the model is stripped rather than failing the whole batch — robustness over rejection. The required-field validation that matters still holds (`evidence_unit_ids.min(1)`, the type enum, nullable defaults).

The run maps each `GeneratedItem` to a `prep_item` (status starts `draft`/`needs_review`), binds `evidence_unit_ids`, then the validator sets the final `status`/`validation_status`/`support_kind`. Items citing an evidence id not belonging to the thesis are rejected by the same-thesis guard (M0d) → that item is recorded `unsafe`, the run continues.

export async function generatePrepPack(
  client: LlmClient,
  args: { thesisId: string; title: string; evidence: { id: string; text: string }[] },
): Promise<GeneratedItem[]> {
  const out = await client.generateObject({
    role: "default",
    purpose: "prep_pack",
    schema: PrepPackSchema,
    prompt: buildPrepPackPrompt(args),
    thesisId: args.thesisId,
  });
  return out.items;
}

```ts
export function createGenerationRun(db: DB, thesisId: string, kind: "prep_pack" | "prep_item" | "regenerate"): string {
  const id = randomUUID();
  db.prepare(
    "INSERT INTO generation_run (id, thesis_id, kind, status) VALUES (?,?,?,'running')",
  ).run(id, thesisId, kind);
  return id;
}
export function finalizeGenerationRun(db: DB, runId: string, status: "done" | "error" | "canceled", error?: string): void {
  db.prepare("UPDATE generation_run SET status=?, error=? WHERE id=?").run(status, error ?? null, runId);
}
export function getThesisEvidence(db: DB, thesisId: string): { id: string; text: string }[] {
  // Thesis reading order (Codex P2-2): order by chunk ord then char span, not by random UUID.
  return db
    .prepare(
      `SELECT eu.id AS id, eu.text AS text
         FROM evidence_unit eu JOIN thesis_chunk tc ON tc.id = eu.chunk_id
        WHERE eu.thesis_id = ? ORDER BY tc.ord, eu.char_start, eu.id`,
    )
    .all(thesisId) as { id: string; text: string }[];
}
export function insertGeneratedPrepItem(
  db: DB,
  item: { thesisId: string; generationRunId: string; type: string; title: string; claim_text: string | null; evidence_quote: string | null; value_numeric: number | null; unit: string | null },
): string {
  const id = randomUUID();
  db.prepare(
    `INSERT INTO prep_item (id, thesis_id, generation_run_id, type, title, claim_text, evidence_quote, value_numeric, unit,
        status, validation_status, validator_version, source)
     VALUES (@id,@thesis_id,@generation_run_id,@type,@title,@claim_text,@evidence_quote,@value_numeric,@unit,
        'needs_review','needs_review','0','generated')`,
  ).run({
    id, thesis_id: item.thesisId, generation_run_id: item.generationRunId, type: item.type, title: item.title,
    claim_text: item.claim_text, evidence_quote: item.evidence_quote, value_numeric: item.value_numeric, unit: item.unit,
  });
  return id;
}
```

```ts
import "server-only";
import type { Database as DB } from "better-sqlite3";
import type { LlmClient } from "./types";
import { generatePrepPack } from "./prep-pack";
import {
  createGenerationRun, finalizeGenerationRun, getThesisEvidence,
  insertGeneratedPrepItem, bindPrepEvidence, getBoundEvidence, applyValidation,
} from "../../db/repository";
import { validatePrepItem, type PrepItemType } from "../evidence/validator";

export async function runPrepPackGeneration(
  db: DB, client: LlmClient, thesisId: string,
): Promise<{ runId: string; runStatus: "done" | "error"; itemCount: number }> {
  const runId = createGenerationRun(db, thesisId, "prep_pack");
  try {
    const evidence = getThesisEvidence(db, thesisId);
    const thesis = db.prepare("SELECT title FROM thesis WHERE id=?").get(thesisId) as { title: string };
    const items = await generatePrepPack(client, { thesisId, title: thesis.title, evidence });
    for (const item of items) {
      const prepId = insertGeneratedPrepItem(db, {
        thesisId, generationRunId: runId, type: item.type, title: item.title,
        claim_text: item.claim_text, evidence_quote: item.evidence_quote,
        value_numeric: item.value_numeric, unit: item.unit,
      });
      // Narrow catch (Codex P1-1): ONLY bindPrepEvidence may fail expectedly (bad/cross-thesis
      // citation). Validator/DB errors are NOT caught here — they propagate to the outer
      // try -> the run is recorded 'error' rather than silently finalized 'done'.
      let bound = true;
      try {
        bindPrepEvidence(db, prepId, item.evidence_unit_ids); // same-thesis enforced (M0d)
      } catch (bindErr) {
        applyValidation(db, prepId, { validationStatus: "failed", supportKind: "existence", reason: bindErr instanceof Error ? bindErr.message : "bind failed" });
        bound = false;
      }
      if (bound) {
        const verdict = validatePrepItem(
          { type: item.type as PrepItemType, claim_text: item.claim_text, evidence_quote: item.evidence_quote, value_numeric: item.value_numeric, unit: item.unit },
          getBoundEvidence(db, prepId),
        );
        applyValidation(db, prepId, verdict);
      }
    }
    finalizeGenerationRun(db, runId, "done");
    return { runId, runStatus: "done", itemCount: items.length };
  } catch (err) {
    finalizeGenerationRun(db, runId, "error", err instanceof Error ? err.message : String(err));
    throw err;
  }
}
```

## Self-Review Notes (author)

- **Validator is the gate, not the generator:** every generated item is inserted `needs_review`, bound to its cited evidence, and only the deterministic validator can move it to `verified`. A hallucinated number/quote → `needs_review`/`unsafe`, never `verified`. This is the P0-2 payoff.
- **Flag for Codex:** the per-item `bindPrepEvidence` failure policy (mark item unsafe, continue the run) — confirm that's preferable to failing the whole run. Also: `insertGeneratedPrepItem` stamps `validator_version '0'`; `applyValidation` overwrites it with the real version — confirm no window where '0' leaks as verified.
- **Deferred:** regeneration/idempotency via `evidence_snapshot_hash`, partial-failure retries, L4 LLM-advisory, per-type prompt specialization, the progress UI.
